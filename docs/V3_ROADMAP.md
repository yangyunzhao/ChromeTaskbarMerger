## 1. V3 定位

V3 是持续性的易用性、美观性和缺陷修复系列，原则上不增加新的产品能力，也不再预先划分一组
长期固定的 Phase。每个彼此内聚的问题或优化形成一个可独立验收、发布和冻结的小版本。

正式版本、PE 版本和 Git 标签使用三段式版本号：口头称“V3.1”，正式产物使用 `3.1.0` 和
`v3.1.0`。按用户决定，V3.1 验收后发现的新 BUG 或质量优化均进入 `3.2.0`，不重新打开 V3.1；
扩大产品能力的需求进入 [V4 功能需求池](V4_BACKLOG.md)。补丁版本只用于已经发布版本的紧急
安全/恢复修复，并需单独得到用户批准。

## 2. 版本归属规则

- 与当前小版本的同一根因、界面区域或恢复路径直接相关：可以加入当前小版本；
- 当前小版本发布前发现的独立 BUG/视觉问题：默认进入下一个 V3 小版本；
- 当前小版本发布后产生的轻微回归通常使用补丁版本；但已按用户决定冻结的 V3.1 新问题进入 V3.2；
- 影响窗口可达、恢复或 Chrome 数据安全的严重问题：可以打断顺序优先修复；
- 会增加新的使用场景、浏览器、窗口组模型或持久化对象：只登记 V4。

每次归属调整都必须写入对应小版本文档，不能无限扩张已经开始验收的版本范围。

## 3. 小版本状态与门禁

每个小版本使用以下状态：

```text
PLANNED
IN PROGRESS
AUTOMATIC PASS / MANUAL PENDING
PASS
BLOCKED
RELEASED
```

每个小版本必须拥有：

- 独立需求文档，记录现象、目标、非目标、安全边界和接受的限制；
- 独立测试记录，保存自动命令、真实环境步骤、结果和恢复方法；
- 至少一个可验证的问题项，不以笼统“优化体验”代替通过标准；
- Debug/Release 和全部既有回归；
- 必要人工验收通过后才允许正式发布。

## 5. V3 共同安全基线

1. 不注入 Chrome、Explorer 或 WindowTabs，不开启远程调试，不修改 Chrome 数据；
2. 不把 Chrome 顶层窗口 `SetParent` 到本程序，不安装全局 Hook、驱动或永久提权组件；
3. 修改任务栏、窗口样式或布局前继续使用写前恢复记录和完整窗口身份校验；
4. 暂停、恢复全部、正常退出、异常结束后的下次启动必须恢复任务栏和组前布局；
5. 内置标签与 WindowTabs 严格二选一；WindowTabs 模式不应用内置标签视觉修改；
6. 任务栏单入口、单实例、登录启动和 profile 名称严格回退不得回归；
7. 视觉方案失败时优先回退 2.0.0 已验证布局，不留下不可达或屏外窗口；
8. 新功能请求记录到 V4，不为赶版本静默扩大范围。
