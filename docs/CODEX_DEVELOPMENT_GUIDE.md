# Codex 开发与验收规范

本文档规定 Codex 在 ChromeTaskbarMerger 仓库中执行开发工作的通用规则。
具体功能范围和技术门槛分别由各版本需求文档定义。V1/V2 使用 Phase；V3 起使用滚动小版本和
问题项，见 [V3 质量改进路线图](V3_ROADMAP.md)。本文后续的“Phase”在 V3 及以后也包含“当前小
版本/问题项”，除非对应版本文档另有更具体规定。

## 1. 指令优先级与适用范围

1. 用户在当前对话中的明确要求优先于本文档。
2. 开发某个版本前，必须完整阅读该版本需求文档和本文档。
3. V1 已发布；除非用户明确要求，不得为了 V2 重写或破坏 V1 的正式标签和发布产物。
4. V2 开发必须持续运行 V1 回归测试，并遵守 V2 的安全不变量。
5. 如果需求文档与当前代码不一致，应先记录差异并确认真实状态，不能只按文档推测。

## 2. 开始一个 Phase 或小版本问题项前

Codex 应完成以下检查：

1. 检查仓库状态、当前分支、现有未提交修改和相关提交历史；
2. 阅读本 Phase 涉及的源码、测试、README 和上一 Phase 记录；
3. 给出简短、可验证的实施计划；
4. 明确哪些项目可以自动验收，哪些项目确实需要用户观察；
5. 为所有可能修改任务栏、窗口位置或启动注册表的操作准备恢复路径；
6. 前一 Phase 未达到通过标准时，不得把后一 Phase 标记为已开始或已完成。

仓库中已有的修改默认属于用户。Codex 必须保留无关修改，不得使用破坏性 Git
命令清除工作区。

## 3. Phase、小版本与问题项状态门禁

每个 Phase 或小版本只能处于以下状态之一：

```text
PLANNED
IN PROGRESS
AUTOMATIC PASS / MANUAL PENDING
PASS
FAIL
BLOCKED
```

- `AUTOMATIC PASS / MANUAL PENDING` 表示自动测试完成，但必要的真实界面观察尚未完成；不能视为正式通过。
- `PASS` 必须满足需求文档列出的全部阻塞性标准。
- `FAIL` 表示已取得明确反例，需要修改设计或实现。
- `BLOCKED` 仅用于缺少必要环境、权限或用户选择且无法安全继续的情况。
- 核心可行性硬门槛失败时，应停止外围功能堆积，先解决或调整技术路线。

## 4. 自动验收优先

Codex 应尽可能自动完成以下工作：

- Debug 和 Release 的全新配置、构建与 CTest；
- `/W4` 警告检查和 `git diff --check`；
- 纯策略、状态机、身份校验、配置解析和恢复日志单元测试；
- 使用伪窗口清单、伪任务栏控制器、可控时钟和故障注入测试边界条件；
- 使用本进程创建的临时顶层窗口完成 Win32 集成测试；
- 使用测试专用临时注册表子键验证登录启动逻辑，并在测试后清理；
- 命令行退出码、便携包清单、PE 架构、子系统、版本资源和动态依赖检查；
- 产物 SHA-256 计算和文档占位符检查。

自动测试必须检查返回值、HRESULT 和 Win32 错误码。API 返回成功不能代替真实任务栏、
窗口层级、焦点或标签切换的视觉验收。

## 5. 何时请求人工验收

只有以下内容无法在当前环境中可靠自动观察时，才请求用户协助：

- Windows 任务栏按钮的真实数量和视觉结果；
- 标签栏是否正确贴合 Chrome、是否闪烁或遮挡；
- 鼠标、键盘、Alt+Tab、最大化、全屏和多显示器的真实交互；
- Explorer 重启、强制结束进程和 Windows 重新登录后的桌面体验；
- 真实 Chrome、真实用户配置文件或特定显示硬件才能复现的问题。

请求人工验收时，Codex 必须一次提供完整说明：

```text
验收目标：
前置条件：
使用的具体 EXE 和版本：
需要执行的命令：
逐步操作：
每一步预期现象：
失败时立即恢复的方法：
需要回传的日志位置：
用户回报模板：
```

操作步骤必须可复制、按顺序执行，并说明是否需要退出旧实例、WindowTabs、Chrome
或 Explorer。不能只要求用户“测试一下”或“看看是否正常”。

## 6. Windows 修改的安全规则

1. 所有任务栏和窗口状态修改必须在明确的单线程模型中串行执行。
2. 修改前必须验证 HWND、PID、TID、进程创建时间和窗口类等身份信息。
3. 写前恢复记录失败时，不得继续执行对应的不可见性、位置或任务栏修改。
4. 恢复操作必须幂等；损坏、截断、重复或超限记录不得被部分采用。
5. 任务栏或标签可达性失败时，优先恢复用户入口，再停止管理。
6. 测试完成、暂停、正常退出和失败回退均应尝试恢复本程序实际修改的状态。
7. 不得注入 Chrome 或 Explorer，不得安装驱动、修改系统文件或修改 Chrome 用户数据。
8. 不得为了测试而关闭用户的 Chrome 窗口；需要关闭时必须明确由用户操作。

## 7. 代码与构建要求

- 生产代码使用 C++20、Unicode Win32 API、CMake 和 MSVC x64。
- 默认警告等级至少 `/W4`，不默认启用 `/WX`。
- Win32 句柄、COM 和其他资源使用 RAII；所有 API 返回值都要处理。
- 不为简单逻辑引入不必要的框架、线程、接口层次或第三方依赖。
- 事件回调只收集信息并投递到主消息循环，不执行长时间阻塞操作。
- 高频事件必须合并或去抖；无状态变化时不重复调用任务栏或窗口 API。
- Release 保持 Windows GUI、x64 和便携运行形态；除非需求变更，不引入管理员权限要求。

## 8. 文档与证据职责

每个 Phase 完成时至少更新：

1. 对应版本需求文档中的 Phase 状态；
2. 对应版本的手工测试计划和实际结果；
3. 用户行为、配置或限制发生变化时更新中英文 README；
4. 便携包行为变化时更新 `docs/PORTABLE_README.md`；
5. 正式发布时增加独立的双语 Release Notes 和最终哈希。

需求文档不重复粘贴完整构建日志。测试记录应保存关键命令、退出码、环境、现象和
结论，并链接到需求条目。

## 9. Git 与发布

- 修改按 Phase 或单一目的组织，避免把多个阶段塞进一个难以审查的提交。
- 提交前检查暂存范围、`git diff --check`、自动测试和工作区状态。
- 构建目录、临时日志和 `dist` 产物不得进入源码提交，除非用户明确改变发布策略。
- 未经用户明确要求，不推送远端、不创建标签、不发布 GitHub Release。
- 正式发布必须使用带注释的版本标签，Release 不得误标为草稿或预发布。
- 发布后应从 GitHub 端复核标签、目标提交、资产、SHA-256、README 和 Release 状态。

## 10. Phase/小版本报告模板

```text
Phase：
实现范围：
修改的主要文件：
自动验收命令与结果：
自动验收覆盖不到的原因：
人工验收步骤与恢复方法：
当前结论：PASS / FAIL / BLOCKED / MANUAL PENDING
遗留问题：
下一 Phase 是否允许开始：是 / 否
```

任何需要人工观察的 Phase 都应在用户回报后记录实际结果，再决定是否进入下一阶段。
