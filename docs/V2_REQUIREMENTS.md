# ChromeTaskbarMerger V2 需求与设计

> 状态：`PASS / RELEASED AS 2.0.0`（2026-07-17）
>
> 设计日期：2026-07-16。本文档基于已发布的 `1.0.0`，定义不依赖 WindowTabs
> 的 V2。开始实现前还应遵守
> [Codex 开发与验收规范](CODEX_DEVELOPMENT_GUIDE.md)。

## 1. 背景与目标

V1 已经可靠完成“多个 Chrome 顶层窗口在 Windows 任务栏只保留一个固定入口”，
但被移除的窗口可能同时从 Alt+Tab 消失，因此必须依靠 WindowTabs 提供顶部标签和
窗口可达性。

V2 的首要目标是完全取消 WindowTabs 运行时依赖，并在同一个便携程序中完成两个
不可分割的核心功能：

1. 把多个 Chrome 顶层主窗口组织成一个由本程序提供的外部多标签界面；
2. 继续使用 V1 已验证的任务栏机制，让这些 Chrome 窗口在 Windows 任务栏只保留
   一个入口。

V2 不是重新开发浏览器。它不把不同 Chrome 配置文件、进程或网页标签真正合并到
Chrome 内部，而是让每个原生 Chrome 顶层窗口对应一个“外部标签”。

## 2. 产品定义

用户看到的目标体验：

```text
ChromeTaskbarMerger 内置标签栏：每个标签对应一个 Chrome 顶层窗口
Chrome 主区域：始终显示当前标签对应的原生 Chrome 窗口
Windows 任务栏：整个组只保留一个 Chrome 入口
WindowTabs：不需要安装或运行
```

组内 Chrome 窗口仍然是独立、原生的顶层 HWND。程序通过统一位置、大小、Z-order
和前台激活，让它们在视觉和交互上表现为同一个多标签窗口。

## 3. V2.0 核心范围

### 3.1 必须实现

- 默认把当前桌面中所有可管理的普通 Chrome 主窗口加入一个自动标签组；
- 每个窗口拥有一个外部标签，至少显示标题、Chrome 图标、活动状态和关闭按钮；
- 鼠标点击标签能够稳定激活正确的 Chrome 窗口；
- 标签切换不关闭、不重启、不注入 Chrome，也不移动 Chrome 内部网页标签；
- 组内窗口共享位置和大小，当前标签窗口位于正确 Z-order；
- 处理窗口新建、关闭、标题变化、最小化、恢复、最大化和基本全屏切换；
- 任务栏始终复用 V1 的 `ITaskbarList::DeleteTab/AddTab` 安全实现；
- 新窗口先获得可用标签，再移除其独立任务栏按钮；
- 标签系统失效时，先恢复任务栏入口，再停止或降级管理；
- 暂停、恢复全部、正常退出和崩溃后重启能够恢复任务栏与窗口布局；
- 保留托盘、日志、单实例、Explorer 重建和随 Windows 登录启动；
- Release 继续保持单目录便携运行，不要求管理员权限。

### 3.2 V2.0 不要求

- 把 Chrome 窗口 `SetParent` 为本程序的子窗口；
- 把不同配置文件真正合并成一个 Chrome 浏览器窗口；
- 控制或重排 Chrome 内部网页标签；
- 实现适用于所有 Windows 应用的通用 WindowTabs 替代品；
- 支持 Edge、Brave、Chromium 或其他浏览器；
- 支持多个用户自定义窗口组；
- 在任务栏提供完整的每标签 DWM 缩略图和代理按钮；
- 完全复制 WindowTabs 的皮肤、插件、拖放和所有快捷键；
- 保证所有 PWA、应用模式、开发者工具和特殊 Chrome 辅助窗口都加入组。

这些功能可以作为 2.1 或更高版本的独立增强，不能阻塞或破坏 V2.0 的核心安全性。

## 4. 已确定的技术方向

### 4.1 参考 WindowTabs 的行为，不形成依赖

WindowTabs 的公开 MIT 源码可作为架构参考，重点研究：

- [`WindowGroup.fs`](https://github.com/leafOfTree/WindowTabs/blob/master/WtProgram/WindowGroup.fs)
  中的组内位置、Z-order、前台、最小化和窗口事件处理；
- [`TabStrip.fs`](https://github.com/leafOfTree/WindowTabs/blob/master/WtProgram/TabStrip.fs)
  中的标签布局和交互；
- [`SuperBarPlugin.fs`](https://github.com/leafOfTree/WindowTabs/blob/master/WtProgram/GroupPlugins/SuperBarPlugin.fs)
  中的任务栏代理与缩略图思路。

V2 生产实现使用 C++20 重新实现所需行为，不引入 WindowTabs 的 F#/WinForms 运行时、
历史 DLL 或针对旧 Explorer 内部结构的 `WtInject` 代码。若直接移植任何可识别源码，
必须保留 MIT 许可证和第三方声明。

### 4.2 使用纯 C++ 原生实现

推荐技术栈：

```text
C++20
Win32
Direct2D / DirectWrite
ITaskbarList / ITaskbarList3（仅在需要时）
SetWinEventHook(WINEVENT_OUTOFCONTEXT)
CMake + MSVC x64
```

选择 C++ 的主要原因是复用现有代码、保持单 EXE 便携部署、减少运行时和内存开销，
以及直接控制 HWND、DPI、DWM、COM 和消息循环。标签切换的主要延迟仍来自 Windows
窗口管理器、DWM 和 Chrome 重绘，不能简单假设语言本身保证零延迟。

### 4.3 不把 Chrome 重父化

禁止使用 `SetParent` 把 Chrome 顶层窗口塞入本程序容器。Chrome 继续保持原生顶层
窗口，本程序只创建自己的标签条弹出窗口，并根据活动窗口更新其 owner、位置和
Z-order。

首选窗口组织策略：

1. 组内 Chrome 使用相同目标矩形；
2. 当前标签对应窗口位于组内最上方并获得前台；
3. 非活动窗口保持打开、非最小化并位于其后，不长期 `SW_HIDE`；
4. 移动或缩放期间如确需临时避让其他窗口，必须先实验验证并保证立即恢复；
5. 标签栏是本程序拥有的独立 Win32 窗口，不改变 Chrome 的 `WS_CHILD/WS_POPUP`。

## 5. 不可破坏的安全不变量

### 5.1 可达性先于任务栏移除

对任意 Chrome 窗口 `W`：

```text
DeleteTab(W) 之前必须满足：
InternalTabExists(W)
&& TabStripHealthy
&& ActivationPathVerified(W)
&& RecoveryIntentPersisted(W)
```

任何窗口没有可工作的内部标签时，其任务栏按钮必须保持可见。

### 5.2 新窗口事务顺序

```text
识别和验证窗口
→ 写入原始布局/恢复意图
→ 创建内部标签
→ 验证点击/激活路径
→ 纳入组内布局
→ 最后调用 DeleteTab
```

任务栏短暂显示两个入口可以接受；先隐藏按钮再创建标签不允许。

### 5.3 失败回退顺序

```text
停止新的 DeleteTab
→ AddTab 恢复所有受影响入口
→ 恢复窗口原始位置和状态
→ 隐藏/销毁标签栏
→ 进入 PausedByError 或 RecoveryRequired
```

恢复失败时不能继续普通管理，也不能销毁唯一仍可访问窗口的路径。

### 5.4 身份与线程

- HWND、PID、TID、进程创建时间和窗口类共同构成窗口身份；
- 每次任务栏、位置、关闭和激活操作前重新验证身份；
- WinEvent 回调只投递事件，所有状态和窗口修改在主 UI 线程串行执行；
- 低频扫描作为事件丢失和 Chrome 特殊行为的安全兜底；
- 不通过注入、驱动、UIAccess、管理员提权或模拟 WindowTabs 鼠标操作实现。

## 6. 推荐架构

```text
ChromeWindowRegistry
        │
        ▼
TabGroupController ───────► TabStripWindow
        │                         │
        ├──────────────────► WindowCoordinator
        │
        ├──────────────────► FixedEntryManager（复用 V1）
        │
        └──────────────────► GroupRecoveryJournal

TrayApplication / ManagementStateMachine 负责总控和故障回退
```

### 6.1 `ChromeWindowRegistry`

- 复用 `chrome_window.*` 的 Chrome 识别和身份信息；
- 初始 `EnumWindows` 建立快照；
- 使用 out-of-context WinEvent 监听创建、销毁、显示、隐藏、标题、位置、最小化和前台事件；
- 合并事件风暴并把稳定变更投递给主线程；
- 保留低频完整扫描，修复漏失事件和句柄复用。

### 6.2 `TabGroupController`

- 保存成员身份、标签顺序、当前活动成员和组状态；
- 确保一个窗口最多属于一个组、一个标签只映射一个有效身份；
- 新建、关闭、切换和主窗口替换采用显式事务；
- 协调标签健康、窗口可达性和任务栏删除资格；
- 不直接绘制 UI，也不直接调用 COM。

### 6.3 `TabStripWindow`

- 使用 Direct2D/DirectWrite 绘制 DPI 感知标签；
- 提供布局、命中测试、悬停、活动状态、关闭按钮和滚动溢出；
- 点击由用户输入触发切换，不通过后台强抢前台；
- 标签栏支持靠左、居中、靠右三种水平对齐；总宽度可配置为组宽度百分比，并按所选边缘
  或中心定位；
- 单个标签宽度可配置并设安全上下限，标签数量超出可用宽度时进入滚动或溢出布局，不无限
  拉长每个标签；
- 默认标签标题来自顶层窗口标题；Phase 6 允许双击标签输入仅在当前程序进程中有效的自定义
  显示名称；Phase 7 在先完成可行性评估并向用户报告后，才考虑按 Chrome profile 持久化；
  V2.0 不承诺无需验证即可准确读取 Chrome profile、Google 账户名称或网页 favicon；
- 标签栏失败或 HWND 被重建时通知总控，不自行继续隐藏任务栏入口。

### 6.4 `WindowCoordinator`

- 计算组矩形和标签栏矩形；
- 使用 `SetWindowPos/DeferWindowPos` 同步位置、大小和 Z-order；
- 处理移动、缩放、最小化、恢复、最大化、Snap、DPI 和基本 F11 全屏；
- 标签点击后激活目标窗口；激活被系统拒绝时记录错误并安全保留访问路径；
- 暂停或恢复时应用保存的 `WINDOWPLACEMENT`，避免窗口留在屏幕外。

### 6.5 `FixedEntryManager`

- 保留 V1 已验证实现和恢复身份规则；
- V2 不重新采用 `WS_EX_TOOLWINDOW` 方案；
- 初期继续保留一个真实 Chrome 固定入口；
- 只有 `TabGroupController` 确认成员可达后才能请求移除；
- 后续任务栏代理窗口属于可选增强，不阻塞 V2.0。

### 6.6 `GroupRecoveryJournal`

在 V1 任务栏恢复信息之外，V2 还需记录：

- 成员完整窗口身份；
- 原始 `WINDOWPLACEMENT` 和可见/最小化/最大化状态；
- 原始显示器和工作区信息；
- 标签顺序和当前活动成员（只作为体验恢复，不作为盲目修改依据）；
- 已完成的任务栏和位置修改步骤。

恢复文件必须版本化，不能破坏 V1 的 `recovery-v1.tsv` 或让旧版本误读 V2 记录。

## 7. V2 状态模型

推荐状态：

```text
Initializing
PreparingGroup
Managing
WaitingForTabProvider
PausedByUser
PausedByConflict
PausedByError
RecoveryRequired
```

状态语义：

- `Initializing`：加载配置、恢复旧状态、枚举窗口；禁止新的 `DeleteTab`；
- `PreparingGroup`：创建标签和同步布局；未就绪成员保持任务栏可见；
- `Managing`：标签组健康且任务栏规则生效；只有此状态允许新的 `DeleteTab`；
- `WaitingForTabProvider`：仅当用户明确选择 `windowtabs` 提供标签、但 WindowTabs 尚未就绪时
  使用；任务栏入口保持可见，就绪后自动进入准备流程；
- `PausedByUser`：用户主动暂停，任务栏和窗口布局已恢复；不会自动恢复管理；
- `PausedByConflict`：内置标签模式检测到 WindowTabs 等外部窗口分组器可能冲突，保持安全可达；
- `PausedByError`：发生错误但恢复已完成，可以由用户重新尝试；
- `RecoveryRequired`：恢复尚未完成，普通恢复管理不得绕过。

`Managing` 的强制不变量：

```text
每一个 removed_from_taskbar 的有效窗口
都存在于健康标签栏中，并拥有经过验证的激活路径。
```

WindowTabs 在默认的内置标签模式中不再是前置条件。只有用户明确选择 `windowtabs` 标签提供
方式时，才允许进入通用的 `WaitingForTabProvider` 状态；不得恢复一个对所有 V2 用户生效的
“等待 WindowTabs”前置条件。

## 8. 默认产品行为

V2.0 默认采用以下简单规则：

1. 自动把同一交互桌面中的所有普通可管理 Chrome 主窗口放入一个组；
2. 首次活动标签优先采用当前前台 Chrome，否则采用稳定发现顺序；
3. 新窗口追加到标签末尾并自动纳入组，但不强制抢占当前前台；
4. 点击标签切换到对应原生窗口；
5. 点击标签关闭按钮重新验证身份后向整个 Chrome 顶层窗口发送正常关闭请求；
6. 标签关闭的是“Chrome 窗口”，不是其中单个网页标签；
7. 暂停管理会恢复任务栏按钮和组前的窗口布局，并隐藏内部标签栏；
8. 恢复管理会重新枚举、创建标签、验证可达性后再应用任务栏规则；
9. 正常退出执行与暂停相同的恢复，再退出托盘；
10. 默认内置标签模式下，WindowTabs 同时运行时进入安全冲突状态并提示用户退出其中一个
    管理器；用户也可在配置或托盘中明确选择 WindowTabs 标签模式，此时不创建内置标签和组
    布局，继续使用已验证的 V1 任务栏单入口能力；两种标签提供方式严格二选一。

V2.0 不提供多组编辑器。多组、拖窗口合组和持久工作区可以后续设计。

## 9. 配置迁移

继续支持：

```ini
[ChromeTaskbarMerger]
scan_interval_ms=2000
start_with_windows=false
tab_provider=builtin
```

`tab_provider` 只接受 `builtin/windowtabs`：

- `builtin` 是 V2 默认值，使用本项目自研外部标签栏、窗口组布局和任务栏单入口，不依赖
  WindowTabs；
- `windowtabs` 不创建本项目标签栏或重排 Chrome 布局，由 WindowTabs 提供标签与窗口可达性，
  本项目继续负责任务栏单入口；WindowTabs 未运行时保持任务栏入口并等待其就绪；
- 该选择可通过托盘的单选菜单或直接编辑 INI 保存。直接编辑以及托盘修改都在完全退出并重启
  后生效，不在受管会话中热切换；
- 从一种模式切换到另一种模式前必须先按当前模式的恢复顺序解除任务栏和布局义务；配置写入
  失败时保持原选择和当前运行模式；
- 旧 V1 配置缺少此键时迁移为 `builtin`，但不删除 `windowtabs_check_interval_ms`，确保仍可降级。

Phase 5 配置扩展：

```ini
[ChromeTaskbarMerger]
tab_strip_alignment=center
tab_strip_width_percent=60
tab_width_px=180
```

其中 `tab_strip_alignment` 只接受 `left/center/right`；百分比和像素值必须经过上下限校验，
非法值回退默认值并记录一次警告。Phase 5 建立的版本化名称规则文件和“进程路径＋窗口类＋
精确标题”匹配只属于安全持久化技术原型，不作为用户自定义名称或 Chrome profile 关联功能的
完成证据。Phase 6 的首次可用编辑只保存在内存；Phase 7 必须先评估可靠的 profile 识别方式，
不得用会在进程重启后失效的 HWND、当前网页标题或不唯一的进程路径冒充 profile 键。

V1 的 `windowtabs_check_interval_ms` 在 V2 中保留为兼容键：

- `tab_provider=builtin` 时保留该值但不启动 WindowTabs 等待定时器；
- `tab_provider=windowtabs` 时作为标签提供器等待/健康检查间隔继续生效；
- 不因旧键存在而拒绝启动；
- 默认不主动删除旧键，便于用户降级到 V1；
- 配置仍只在完全退出并重启后生效，不实时监控。

只有经过实际需求验证的标签高度、主题或快捷键才加入配置，不能在核心可行性验证前
堆积外观选项。

## 10. 分阶段开发与验收计划

所有 Phase 遵守以下共同规则：

- V1 现有自动测试必须持续通过；
- 集成模式下不得移除 V1 的任务栏单入口能力；
- 第一个用户可见的 V2 硬门槛同时验收内部标签和任务栏单入口；
- 每个修改窗口或任务栏的测试都必须提供恢复路径；
- 自动验收完成但必要视觉测试未完成时，只能标记
  `AUTOMATIC PASS / MANUAL PENDING`；
- 详细结果写入 Phase 0 新建的 `tests/manual_test_plan_v2.md`。

### Phase 0：V1 基线保护与 V2 测试骨架

状态：`PASS`

目标：在不改变已发布用户行为的前提下建立 V2 模型、测试接缝和独立验收记录。

实现内容：

- 新建 `tests/manual_test_plan_v2.md`；
- 记录当前 V1 Debug/Release、CTest、便携包和恢复基线；
- 为窗口注册器、标签组、窗口协调器和任务栏网关定义最小职责边界；
- 引入不依赖真实 Chrome 的窗口身份和标签组纯模型；
- 为伪任务栏控制器、伪激活器、可控时钟和故障注入准备测试接缝；
- 保持默认程序仍可构建和运行，不删除 `FixedEntryManager` 或恢复能力。

Codex 自动验收：

- 全新 Debug/Release 无警告构建；
- V1 全部 CTest 继续通过；
- 新增模型测试覆盖空组、1/3/5 个成员、重复添加、关闭、句柄身份变化和顺序稳定；
- `git diff --check`、命令行和便携构建基线通过；
- 证明本 Phase 没有调用新的窗口位置或任务栏修改路径。

人工验收：通常不需要。

通过标准：V1 行为和测试零回归，V2 核心模型可测试，独立测试记录建立。Phase 0
不得以删除或禁用任务栏功能换取简化。

## 11. V2.0 最终验收标准

1. 未安装或未运行 WindowTabs 时程序可以正常启动和管理；
2. 1、3、5 个可管理 Chrome 窗口分别显示对应数量的内部标签；
3. 存在 Chrome 时 Windows 任务栏恰好保留一个 Chrome 入口；
4. 每个被任务栏移除的窗口都可通过内部标签稳定到达和操作；
5. 新建、关闭、全部关闭和重开后标签数和任务栏状态自动收敛；
6. 标签切换不改变 Chrome 配置文件、网页标签、下载、扩展或用户数据；
7. 移动、缩放、最小化、恢复、最大化和基本 F11 不留下屏外或不可达窗口；
8. 用户暂停、恢复全部和正常退出恢复任务栏及组前窗口布局；
9. 强制结束后的恢复不依据陈旧身份修改无关窗口；
10. Explorer 重建后托盘、标签和任务栏规则能够安全恢复；
11. 随 Windows 登录启动无需等待 WindowTabs；
12. WindowTabs 同时运行时不会与本程序静默竞争导致窗口不可达；
13. 空闲 CPU 接近 0%，内存和 USER/GDI 句柄不存在持续增长；
14. Debug/Release、自动测试、便携包、版本资源和恢复文档全部通过；
15. 最终真实环境结果由用户确认，未执行的显示矩阵明确记录。
16. 双击内置标签可输入包含中文的自定义名称，当前程序进程内稳定有效；持久化默认关闭时重启后
    恢复实时标题，显式启用后仅为唯一验证的普通本地 profile 恢复名称；
17. profile 关联持久化在编码前有明确可行性结论和用户确认；若批准实现，不同 profile 不串用
    名称；若不实现，产品和文档明确保留仅内存降级，不伪造关联。

## 12. 已知技术风险

| 风险 | 设计应对 |
| --- | --- |
| Windows 拒绝后台 `SetForegroundWindow` | 只在用户点击/快捷键输入路径激活；失败时保留入口并记录 |
| Chrome 快速重建或复用 HWND | 完整窗口身份、事件合并和扫描兜底 |
| DPI、Snap 和负坐标导致窗口屏外 | Per-Monitor V2、统一几何层和工作区恢复校验 |
| 标签 UI 崩溃后隐藏窗口不可达 | 可达性门禁；先 AddTab，再恢复布局和停止管理 |
| 部分事务后进程被强制结束 | 版本化写前日志和逐步骤恢复 |
| WindowTabs 同时操作相同窗口 | 冲突检测、恢复后暂停和明确提示 |
| 管理员权限 Chrome 受 UIPI 限制 | 检测失败并排除，不自动请求提权 |
| 模态对话框或 F11 改变焦点/层级 | 专项事件和真实 Chrome 人工回归 |
| Direct2D 设备丢失或 Explorer 重建 | 可重建设备资源；标签失败不影响任务栏恢复 |
| Windows/Chrome 更新改变行为 | 保留诊断日志、扫描兜底、身份校验和阶段回归 |
| Win32 窗口无法可靠映射 Chrome profile | 已采用 User Data、UIA 唯一名称和创建时间验证链；任一步失败或歧义时仅使用内存名称并回退实时标题 |

## 13. 明确禁止的实现

- 把 Chrome 顶层窗口 `SetParent` 到本程序；
- 注入 Chrome、Explorer 或 WindowTabs；
- 使用 Explorer 未公开内存结构或全局 DLL Hook；
- 安装内核驱动、修改系统文件或要求永久管理员权限；
- 修改 Chrome 用户数据、配置文件目录、快捷方式或启动参数；
- 把不同配置文件的网页标签移动到同一个 Chrome 配置上下文；
- 长期 `SW_HIDE`、频繁最小化或移到不可恢复位置来伪装标签切换；
- 在内部标签未创建或不可激活时隐藏对应任务栏按钮；
- 只凭 HWND 恢复位置、关闭窗口或修改任务栏；
- 用模拟鼠标点击 WindowTabs 作为 V2 实现；
- 为追求外观而绕过恢复门禁或忽略 API 错误。
