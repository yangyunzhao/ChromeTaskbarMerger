# ChromeTaskbarMerger V3.1 顶部视觉融合与原生最大化

> 正式版本目标：`3.1.0`
>
> 状态：`PASS / FROZEN`
>
> 开始日期：2026-07-17
>
> 完成日期：2026-07-18

## 1. 问题与目标

2.0.0 的内置标签栏作为独立矩形窗口占用 Chrome 上方一整条区域。受管最大化会先接收原生
最大化操作，再把 Chrome 还原为 Windows 认为的普通状态并手工铺满工作区下部，因此存在三项
相互关联的体验问题：

1. 普通窗口的标签栏与 Chrome 视觉割裂，不如 WindowTabs 的紧凑顶部吸附效果；
2. 最大化时标签栏没有进入 Chrome 顶部空闲区域，额外占用垂直空间；
3. Chrome 实际处于普通状态，右上角仍显示“最大化”图标，虽然点击语义被程序解释为还原。

V3.1 将这三项视为同一个优化：**让 Chrome 在最大化时保持真实原生最大化状态，并把内置窗口
标签叠加到 Chrome 顶部；普通状态则使用紧凑的顶部吸附外观。** 如果原生最大化方案成立，
最大化/还原图标应由 Windows/Chrome 自然正确绘制，不单独伪造图标。

## 2. 已确认的视觉参考

用户于 2026-07-17 提供三张本地参考图：

- 图 1：WindowTabs 在普通窗口下以一组紧凑标签吸附在 Chrome 上边缘并靠右呈现；
- 图 2：Windows 窗口最大化时，WindowTabs 标签进入 Chrome 顶部靠右空闲区域，与原生网页标签
  共用一行，并避让右上角三个窗口按钮；
- 图 3：Chrome 原生网页标签接近约 10 个后，WindowTabs 标签直接覆盖右侧原生标签。

图 3 的高数量遮挡由用户明确接受。V3.1 不探测 Chrome 原生标签末端，不实现碰撞检测、折叠、
压缩或动态让位。“约 10 个”只描述参考图，实际阈值会随窗口宽度、DPI 和标题宽度变化。

参考图片当前只作为本地设计输入；未经用户明确许可，不将包含网页标题的图片提交到 Git。

## 3. 期望行为

### 3.1 普通窗口

- 标签组紧贴 Chrome 窗口上边缘并靠右呈现，不显示贯穿全宽的独立工具栏背景；
- 标签外观使用紧凑圆角、清晰的活动/非活动状态、图标、文字和关闭按钮间距；
- 拖动或缩放任一组成员时，标签组稳定跟随，松开鼠标后不跳动；
- 保留 2.0.0 的 `left/center/right`、总宽度百分比、单标签宽度和滚轮溢出兼容性；参考视觉
  以 `right` 为基线；
- 不能安全绘制新外观时回退 2.0.0 外部顶部标签栏。

### 3.2 Windows 原生最大化

- 全部受管 Chrome 保持 Windows 原生最大化状态，`IsZoomed`/`WINDOWPLACEMENT` 与显示状态一致；
- 内置标签窗口叠加在 Chrome 顶部靠右区域，不额外占用一行，不改变 Chrome 最大化工作区矩形；
- 标签区域右边缘必须避让最小化、最大化/还原和关闭按钮；
- Chrome 地址栏和三个窗口按钮始终可操作；原生网页标签很多时允许右侧部分被覆盖；
- 原生最大化状态应自然显示“还原”双框图标，不绘制假的标题按钮；
- 点击任意受管 Chrome 的还原按钮后，整个组恢复到最大化前的统一布局；
- 最小化后恢复时仍回到原生最大化组，图标、标签层级和任务栏入口正确；
- 标签窗口失效、置顶或定位失败时恢复所有任务栏入口/布局义务并安全暂停，或回退已验证布局。

### 3.3 Snap、Explorer 与提供器

- Snap 保留 2.0.0 行为和标签位置稳定性，不要求采用最大化融合布局；
- Explorer 重建后托盘、任务栏入口和标签层级能重新建立；
- 暂停、恢复全部、正常退出和异常恢复不残留叠加标签、窗口样式或错误 placement；
- `tab_provider=windowtabs` 不创建或应用任何 V3.1 内置标签窗口；
- F11 真正全屏属于明确非目标，不设计、不实现、不加入自动或人工验收矩阵。

## 4. 非目标与接受的限制

- 不解决 Chrome 原生网页标签与内置窗口标签在高数量下的重叠；
- 不增加侧边/底部标签、自由拖动标签栏、多套布局或新主题系统；
- 不复制、反编译、注入或自动操作 WindowTabs；
- 不修改 Chrome 原生网页标签数据、窗口标题、profile 或启动参数；
- 不保证 Windows 10、其他 Chromium 浏览器及所有多显示器/DPI 组合；
- 不为了视觉相似使用不可恢复的窗口样式、全局 Hook 或固定像素硬编码。

## 5. 技术方向与失败门禁

首选路径：在最大化状态保留 Chrome 原生 `SW_MAXIMIZE`/placement，让 Chrome 自然绘制还原图标；
标签栏使用不激活的顶层窗口，随当前活动成员保持正确 Z 序并只占用顶部安全区域。

编码前后的可行性检查必须证明：

1. 多个受管 Chrome 可以同时保持原生最大化，切换活动成员时没有持续抢焦点或层级循环；
2. 标签窗口能够位于当前 Chrome 之上而不遮挡窗口控制按钮，也不会被非活动成员永久盖住；
3. 原生还原事件可以被现有事件合并机制观察，并原子恢复整个组；
4. 最小化/恢复、Explorer、暂停和异常恢复能精确恢复原 placement；
5. 任何一步失败时不隐藏最后可达入口，不留下无法退出的叠加窗口。

若首选路径无法满足这些门禁，不得退回“伪造还原图标”。应保留 2.0.0 最大化布局，记录证据并
与用户讨论降低目标或停止 3.1.0。

问题项是验收清单，不是必须串行的 Phase；相关实现可以一起修改，但每项都必须独立记录结果。

2026-07-18 首轮真实 Chrome 验收确认功能无异常，但图 4、5 显示 GDI + `SetWindowRgn` 版本的
逐标签圆角存在明显整数像素阶梯，标签偏高、边框偏重且整体不够精致。该问题与 V3.1 的顶部视觉
主题直接相关，因此不另开新版本：候选改为 Direct2D 曲线路径、DirectWrite 字体、WIC 图标缩放、
矢量关闭符号和 DWM 外层圆角，并移除逐标签一位 region 裁剪；人工视觉复验通过前 V3.1-B/D
不得标记完成。

图 6、7 的第二轮真实 Chrome 复验确认抗锯齿、字体、图标、尺寸和整体精细度已明显改善，但图 1
对比仍暴露两个合成差异：相邻标签之间会露出不透明窗口底板和底边横线；普通窗口右侧还会包含
DWM 外层阴影，导致可见范围越过 Chrome 右边缘。V3.1 继续使用逐像素透明 `WS_EX_LAYERED` 表面，
以透明像素清空非标签区域；标签轮廓改为 Chrome 风格肩部曲线并取消逐标签底边描边，同时禁止
DWM 再为透明顶层窗口添加外层圆角和非客户区阴影。最终外观仍以人工截图复验为准。

图 8、9 证明上述第二修正版整体反而弱于图 6、7：肩部曲线过强并形成尖角/挤压，取消全部阴影后
层次感下降；右侧仍超出也证明根因不是标签窗阴影，而是 Chrome 的 `GetWindowRect` 包含不可见
缩放边框。该轮轮廓方案被否决。第三修正版以图 6、7 的紧凑圆角轮廓为基准，保留逐像素透明和
无底边描边，只加入克制的内部半透明阴影；横向边界使用 Chrome 的
`DWMWA_EXTENDED_FRAME_BOUNDS` 与外部窗口矩形之差修正，最大化按钮保留区不重复扣减。

图 10、11 的最终真实 Chrome 复验确认第三修正版明显优于图 8、9：普通窗口的透明标签间隙、
底边、轻阴影和 Chrome 可见右边界均自然；最大化标签与 Chrome 顶部融合且窗口按钮保留区正确。
用户于 2026-07-18 明确批准 V3.1 到此完成。此后发现的新 BUG 或优化统一进入 V3.2，不再修改
V3.1 已验收视觉基线；本地参考截图仍不进入 Git。

## 7. 自动验收

- 纯布局模型覆盖普通/最大化/Snap、负坐标、100%～200% DPI、1/3/5 个内置标签和溢出；
- 原生状态模型覆盖普通→最大化→还原、最大化→最小化→恢复、标题拖动还原和成员切换；
- 本进程临时顶层窗口验证多个原生最大化成员、还原图标对应的真实 `IsZoomed`、标签 Z 序和
  控制按钮保留区；
- 窗口事件合并不产生忙循环、持续前后台切换、无界队列或重复任务栏调用；
- 暂停、恢复全部、正常退出、Explorer 重建和强制结束恢复精确回滚 placement、任务栏和标签窗口；
- WindowTabs 提供器隔离、配置迁移、profile 名称、登录启动及 V2 全部回归通过；
- 全新 x64 Debug/Release `/W4`、全部 CTest、PE/manifest、静态运行库、便携清单和 SHA-256 通过。

自动测试不把截图审美相似度伪装成结果，也不增加 Chrome 原生网页标签碰撞测试。

## 8. 必要人工验收

使用最终候选和真实 Chrome 验证：

1. 1、3、5 个普通窗口的顶部吸附、切换、拖动、缩放和滚轮溢出；
2. 最大化后与图 2 类似地融入顶部，右上角显示原生还原图标且三个窗口按钮无遮挡；
3. 点击还原、最小化/恢复、任务栏单入口、切换成员和标题区拖动行为；
4. 原生网页标签较少时布局舒适；约 10 个或更多时允许图 3 所示遮挡，不判失败；
5. 暂停/恢复、Explorer 重建、强制结束后的下次启动恢复、正常退出和 WindowTabs 模式；
6. 空闲 CPU、内存、USER/GDI 对象和焦点/闪烁观察。

完整步骤、实际结果和恢复方法记录到 `tests/manual_test_plan_v3_1.md`。

## 9. 通过与发布标准

- V3.1-A～E 均有自动证据，必需视觉项目由用户确认；
- 最大化保持真实原生状态并自然显示还原图标，或明确判定首选路径不可安全实现且停止发布；
- 不存在窗口不可达、窗口按钮遮挡、无法恢复、持续闪烁/焦点循环或 Chrome 数据影响；
- V2 已有能力和 WindowTabs 模式无回归；
- 用户已明确批准 `3.1.0` 代码与文档基线；Git 标签和非预发布 GitHub Release 仍是独立的显式
  发布步骤，不由本次提交/推送自动创建。
