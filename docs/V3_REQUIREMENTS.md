# ChromeTaskbarMerger V3 易用性、视觉完善与缺陷修复计划

> 状态：`PLANNING`
>
> 规划日期：2026-07-17
>
> 基线版本：[`2.0.0`](https://github.com/yangyunzhao/ChromeTaskbarMerger/releases/tag/v2.0.0)

## 1. 版本定位

V3 是一次以**易用性、美观性和缺陷修复**为中心的质量版本。原则上不增加新的产品能力，
不改变 V2“内置标签或 WindowTabs 二选一、多个 Chrome 窗口组成一组、任务栏只保留一个
Chrome 入口”的产品边界。

V3 可以为了修复现有界面或表达现有行为而调整内部模型、窗口样式、布局策略、配置迁移和
诊断日志，但不能借机加入新的使用场景。开发中收到的新功能需求统一登记到
[V4 功能需求池](V4_BACKLOG.md)，不进入 V3 实现和发布门禁。

## 2. 范围判断规则

### 2.1 V3 范围内

- 已有按钮、标签栏、窗口组或托盘行为的错误修复；
- 已有操作难以发现、反馈不清、视觉割裂、遮挡、闪烁、焦点异常或布局不稳定；
- 使现有标签栏在普通、最大化、Snap、F11、DPI 或 Explorer 重建后保持正确位置；
- 使现有最大化/还原行为的图标、视觉状态和点击语义一致；
- 为上述修复增加必要的诊断日志、测试接缝和保守配置迁移。

### 2.2 必须移入 V4

- 新增浏览器、多个独立窗口组、工作区、标签拖放合组/拆组或新的持久化对象；
- 新增任务栏缩略图模型、浏览器网页标签管理、账户管理或跨设备同步；
- 新增一整套主题、插件、脚本、自动化规则或与本轮缺陷无关的快捷键体系；
- 任何会扩大用户可完成任务范围，而不只是修正现有体验的需求。

边界有争议时，默认先登记 V4；只有用户确认其属于 V3 的现有体验修复后才能移入 V3。

## 3. V2 安全与兼容基线

V3 的视觉改动不得降低以下已经通过的 V2 能力：

1. 不注入 Chrome、Explorer 或 WindowTabs，不开启远程调试，不修改 Chrome 数据；
2. 不把 Chrome 顶层窗口 `SetParent` 到本程序，不安装全局 Hook、驱动或永久提权组件；
3. 每次修改任务栏、窗口样式或布局前仍遵守写前恢复记录和完整窗口身份校验；
4. 暂停、恢复全部、正常退出、异常结束后的下次启动均能恢复任务栏和组前布局；
5. 内置标签和 WindowTabs 保持严格二选一，WindowTabs 模式不应用 V3 内置标签视觉修改；
6. 1～5 个受管 Chrome、任务栏单入口、单实例、登录启动和 profile 名称回退不得回归；
7. Chrome 原生网页标签、地址栏、最小化/最大化/关闭按钮和 F11 始终可达；
8. 任何视觉方案失败时，优先回退到 V2 已验证布局，不能留下不可达或屏外窗口。

## 4. 已确认的 V3 需求

### 4.1 受管最大化按钮图标专项

#### 当前问题

V2 受管最大化的点击行为正确：再次点击中间按钮可以还原窗口组，但 Chrome 右上角仍可能显示
“最大化”图标，而不是传统的“还原”图标。图标与真实操作语义不一致，降低了可理解性。

#### 设计要求

- 先确定图标由 Chrome 非客户区状态、`WINDOWPLACEMENT`、窗口样式、DWM 或 Chrome 自绘状态中
  的哪一层决定，并记录最小复现；
- 只使用公开、可回滚且能够精确恢复的 Win32/DWM 能力；
- 图标、真实 `IsZoomed`/placement 状态和点击语义必须一致，不能只绘制一个假的还原图标；
- 不得为了改变图标破坏 Snap、F11、系统菜单、双击标题区或组布局；
- 若 Chrome 当前实现无法在安全边界内控制该图标，必须提供证据和明确限制，保留 V2 行为，
  不使用高风险绕过方案。

#### 验收标准

- 自动：覆盖普通→最大化→还原、最大化→最小化→恢复、左右 Snap、F11、Explorer 重建、异常恢复
  和原始样式精确恢复；
- 人工：确认图标与行为一致，Chrome 原生三个标题按钮和网页标签均可用，任务栏单入口、组移动和
  正常退出无回归；
- 若结论为无法安全修正，技术评估、复现证据和用户接受的限制说明构成本项验收结果，但不得静默
  宣称已经修复。

### 4.2 标签栏位置

#### 当前问题

V2 的标签栏始终作为组顶部的外部窗口存在，已有 `left/center/right` 水平对齐、总宽度百分比和
单标签宽度配置，但“标签栏位置”仍缺少完整的产品定义。普通、最大化、Snap 和不同 DPI 下，
标签栏应锚定到哪里、是否占用额外高度、与 Chrome 原生顶部区域保持多少间距，需要统一。

#### V3 定义门禁

编码前必须由参考图或标注截图确认以下内容：

- 普通窗口和最大化窗口各自的目标位置；
- “位置”是指水平锚点/偏移、窗口上边缘关系，还是需要进入 Chrome 顶部的可用空白区域；
- 标签栏是否占用独立的一行，是否允许覆盖 Chrome 区域，以及必须避开的原生按钮和拖动区域；
- 1、3、5 个标签和溢出时的期望宽度、起点和滚动表现；
- 单显示器基线以及负坐标、不同 DPI、工作区变化时的锚定规则。

V3 只实现为修正现有标签栏位置所必需的策略。任意自由拖放标签栏、多套可保存布局或与窗口组
无关的悬浮面板属于新功能，登记 V4。

#### 验收标准

- 自动：布局策略使用纯数据输入覆盖普通、最大化、左右 Snap、100%～200% DPI、负坐标、1/3/5
  标签和溢出；输出不得越出工作区或侵入已声明的 Chrome 原生保留区域；
- 人工：按最终确认的参考图比较普通与最大化状态，拖动/缩放期间位置稳定，松开鼠标后无跳动，
  标签可点击且不遮挡 Chrome；
- 配置迁移必须保留 V2 合法值，非法或已废弃值回退安全默认并只记录一次说明。

### 4.3 最大化时标签栏融入 Chrome 顶部

#### 目标体验

最大化后，内置标签栏不应像一个与 Chrome 割裂的额外浮动窗口，也不能遮挡 Chrome 自带网页
标签或最小化/最大化/关闭按钮。目标是获得与 WindowTabs 类似的统一窗口观感：用户感知到一个
完整窗口，能够同时使用 Chrome 原生控件和 ChromeTaskbarMerger 的窗口标签。

“与 WindowTabs 类似”目前只是体验方向，不代表复制其实现。**进入实现前需要用户提供参考图片，
或由用户确认 Codex 制作的带标注示意图。** 至少需要一张 WindowTabs 最大化效果图；如普通窗口
效果也不同，建议同时提供普通状态图片。

#### 可行性研究要求

- 分别评估“工作区外部紧贴”“非客户区安全叠加”“Chrome 顶部空白区域视觉融合”等公开 API
  方案，不预设必须修改 Chrome 非客户区；
- 识别 Chrome 原生网页标签区、窗口控制按钮、可拖动区域和系统保留区域，任何方案都不得依赖
  固定像素猜测；
- DPI、Chrome UI 密度、窗口宽度、标签数量和系统文本缩放变化时仍应可计算；
- 标签栏窗口失效或定位失败时立即回退 V2 外部顶部布局，并保持所有 Chrome 可达；
- 不复制、反编译或注入 WindowTabs；WindowTabs 只作为经用户确认的视觉和交互参考。

#### 验收标准

- 最大化时没有额外空白缝隙、双层边框、明显悬浮感或遮挡；普通↔最大化切换过程无持续闪烁；
- Chrome 原生网页标签、地址栏和三个窗口按钮可完整点击，内置标签也可切换、滚动和改名；
- 在窄窗口、1/3/5 标签、125%/150%/200% DPI、任务栏工作区变化、Snap、F11 和 Explorer 重建
  后仍安全；
- 暂停、切换 WindowTabs、恢复全部、正常退出和异常恢复后不残留叠加窗口或样式；
- 最终视觉必须由用户人工确认。几何与可达性尽量自动验收，审美相似度不伪装成自动测试结果。

## 5. 缺陷修复与易用性收集方式

V3 开发期间允许继续加入缺陷和易用性问题，但每项必须记录：

```text
标题：
类型：BUG / USABILITY / VISUAL / FEATURE-CANDIDATE
环境：Windows、Chrome、DPI、显示器、标签提供器
复现步骤：
期望结果：
实际结果：
截图或日志时间：
是否影响窗口可达或恢复：
归属：V3 Phase / V4_BACKLOG
```

影响窗口可达、恢复错误、Chrome 原生控件不可用或数据安全的问题优先级最高。纯审美差异不得以
破坏安全恢复为代价修复。`FEATURE-CANDIDATE` 默认只登记到 V4。

## 6. 开发 Phase

所有 Phase 使用 `NOT STARTED / IN PROGRESS / AUTOMATIC PASS / MANUAL PENDING / PASS / BLOCKED`
状态，并遵守 [Codex 开发与验收规范](CODEX_DEVELOPMENT_GUIDE.md)。Phase 0 开始时建立
`tests/manual_test_plan_v3.md`，后续实际命令、日志、图片说明和用户回报只写入测试记录。

### Phase 0：V2 基线、视觉规范与诊断接缝

状态：`NOT STARTED`

目标：冻结 2.0.0 行为基线，把三个需求从文字方向转化为可比较的状态、几何和视觉规范，本 Phase
不改变正式窗口行为。

实现与自动验收：

- 全新 Debug/Release、全部 CTest、2.0.0 便携恢复基线通过；
- 建立普通/最大化/Snap/F11 状态矩阵和标签栏几何快照模型；
- 增加只读诊断，记录窗口 placement、样式、DWM 属性、DPI、工作区和标签栏目标矩形，不记录
  网页标题、profile 名称或用户自定义名称；
- 建立最大化图标最小复现和布局策略测试接缝，不修改真实 Chrome 状态。

人工验收：用户提供或确认 WindowTabs 普通/最大化参考图，并确认标签栏位置的含义、占用区域和
1/3/5 标签目标示意。

通过标准：V2 零回归；视觉参考和位置语义不存在阻塞歧义；后续 Phase 有可自动断言的输入输出。

### Phase 1：最大化状态与按钮图标一致性

状态：`NOT STARTED`

目标：完成图标来源研究，在不破坏窗口状态和恢复的前提下使图标与最大化/还原语义一致。

自动验收：状态转换矩阵、Snap/F11、最小化恢复、Explorer 重建、异常恢复、样式/placement 精确
回滚及 V2 全量回归通过；不可安全实现的分支必须失败关闭并回退原行为。

人工验收：用户观察图标、点击行为、系统菜单、双击顶部区域和 Chrome 原生按钮。

通过标准：安全修正并由用户确认；或者形成证据充分的不可行结论，经用户接受后明确保留限制。

### Phase 2：标签栏定位模型与普通窗口完善

状态：`NOT STARTED`

目标：依据 Phase 0 视觉规范统一标签栏锚点、间距、工作区约束和 DPI 计算，先保证普通、移动、
缩放和 Snap 状态稳定，为最大化融合提供同一布局模型。

自动验收：合成显示矩阵、1/3/5 标签、溢出、左右/中间对齐、边界约束、DPI 变化和事件风暴通过；
位置更新无忙循环、无无界队列、无持续来回纠偏。

人工验收：拖动任一成员、快速斜向移动、缩放、左右 Snap、最小化恢复，确认标签栏无明显跳动、
漂移、遮挡或点击错位。

通过标准：普通与 Snap 状态达到确认的目标位置，原配置安全迁移，暂停/退出恢复无回归。

### Phase 3：最大化标签栏视觉融合

状态：`NOT STARTED`

目标：按 Phase 0 已确认参考图完成最大化视觉融合；不能安全融合的环境自动回退 Phase 2/V2 外部
顶部布局。

自动验收：目标区域计算、原生保留区避让、DPI/宽度/标签数量矩阵、普通↔最大化动画期事件、F11、
Snap、Explorer、提供器切换、暂停/恢复和故障回退通过。

人工验收：用户对照参考图确认整体观感，并验证 Chrome 网页标签、地址栏、三个窗口按钮、内置标签
切换/滚动/改名全部可用。

通过标准：视觉由用户确认；不存在遮挡、窗口不可达、恢复残留或持续闪烁；不支持环境能明确且
安全地降级。

### Phase 4：V3 缺陷与易用性收口

状态：`NOT STARTED`

目标：集中修复 Phase 1～3 实测发现的 BUG、USABILITY 和 VISUAL 问题，不接收新功能实现。

自动验收：每个修复先有可复现测试或诊断证据；全量 Debug/Release、压力事件、空闲资源、恢复
故障注入和配置迁移通过。

人工验收：只复测实际发生过的真实界面问题和 V3 核心路径；新功能建议登记 V4。

通过标准：阻塞/高优先级缺陷为零；其余已知限制有复现条件、影响和安全回退说明。

### Phase 5：3.0.0 发布候选与正式发布

状态：`NOT STARTED`

目标：形成普通用户可直接使用的 3.0.0 正式便携包。

自动验收：全新 x64 Debug/Release `/W4`、全部 CTest、PE/manifest/版本资源、静态运行库、便携清单、
ZIP 和 SHA-256 通过；中英文 README、便携说明、V3 测试结果和双语 Release Notes 完整；真实 Run
项、恢复日志和测试进程无残留。

人工验收：使用最终 ZIP 抽查普通/最大化视觉、按钮图标、标签栏位置、1/3/5 窗口、暂停/恢复、
Explorer、异常恢复、登录启动、WindowTabs 模式和正常退出。

通过标准：Phase 0～4 均为 PASS；用户明确批准后才提交发布标签并创建非预发布 GitHub Release。

## 7. V3 最终验收标准

1. V2 已有功能和安全恢复能力无回归；
2. 最大化按钮图标与真实状态和点击语义一致，或有用户接受的安全不可行结论；
3. 标签栏在普通、最大化、Snap、F11 和 DPI 变化后位于已确认目标位置；
4. 最大化视觉达到用户确认的 WindowTabs 类似融合效果，或在不支持环境安全降级；
5. Chrome 原生网页标签、地址栏及三个窗口按钮始终无遮挡且可操作；
6. 不产生持续闪烁、焦点抖动、位置来回纠偏、忙循环或资源持续增长；
7. 暂停、恢复全部、退出、Explorer 重建和异常恢复不残留标签窗口、窗口样式或任务栏状态；
8. WindowTabs 模式不受内置标签视觉修改影响；
9. 新功能请求均已移入 V4，没有未经确认的范围扩张；
10. 最终自动结果、必要人工结果、限制和参考图说明完整记录。

3. 用一句话说明“标签栏位置”主要是指左右/居中和偏移，还是顶部/侧边/底部，或者就是最大化时
